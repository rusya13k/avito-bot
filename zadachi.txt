КОНТЕКСТ И ЦЕЛЬ
================

Текущее состояние: бот функционально готов. Sprints 1-5 + E2 + G1 + G2 done.
84 тестов проходят, ruff чистый. Селениум-флоу за единым фасадом
AvitoClient, метрики per account / per hour работают, accounts.json вынесен.

Реальная проблема СЕЙЧАС — лайфтайм Avito-аккаунтов. Без описанных ниже
доработок реалистичная вилка:

  • proxies.txt пуст (общий IP)        → 3-7 дней до капчи, 1-2 нед до бана
  • резидентский proxy per account     → 2-3 недели до проблем, 1-2 мес до бана
  • + дневные лимиты + warmup          → 1-3 мес до проблем, 3-6+ мес до бана

ВЫВОД: главный приоритет сейчас — не новые фичи, а защита аккаунтов.

Что УЖЕ работает в защиту:
  • AdsPower (antidetect fingerprinting) — главное
  • B2 human-delays (normal distribution в ключевых точках)
  • B4 native session reuse (после первого логина не дёргаем /login)
  • A3 captcha-cooldown 30 мин (per-account override через G2)
  • B1 SMS/captcha обработка через TG-резюм
  • Random sample(5) + random city/category/deal_type на каждую итерацию
  • Yandex warmup (не заходим прямо на Avito)
  • E2 metrics — есть на чём строить дневные лимиты

Что НЕ защищено и палит:
  • proxies.txt пустой
  • Нет дневных лимитов — поток крутится while True
  • Клик "Показать телефон" на каждом листинге без капа (самое палевное)
  • Нет cooldown'а между прогонами (через 5 сек после messenger начинаем search)
  • Нет warmup для нового аккаунта (сразу клацает на телефоны)
  • Парсит 24/7 без time-of-day pattern
  • Все потоки стартуют с фиксированным time.sleep(5) — регулярный паттерн

БЛОК A — TIER 1: ЗАЩИТА АККАУНТОВ (без этого бан за дни)

A1. Per-account proxy [M]

  • Файлы: accounts.py, bot.py (run_thread), accounts.example.json
  • Проблема: proxies.txt пуст. Если у тебя 2+ аккаунта — все идут с
    одного IP, Avito склеит их в одного пользователя и забанит вместе.
  • Шаги:
    1. Добавить поле proxy в схему accounts.json:
       "proxy": "user:pass@host:port" или "socks5://user:pass@host:port"
    2. В bot.run_thread: если в account есть proxy — обновить AdsPower-
       профиль перед start_profile через update_profile_proxy(adspower,
       user_id, proxy) (функция уже есть в bot.py:449)
    3. Если proxy отсутствует / невалиден — fallback на get_random_proxy()
       из proxies.txt; если и там пусто — ERROR в TG (E4) и пометить
       аккаунт enabled=false на этот запуск
    4. Документировать в README, что для прод-использования нужны
       residential / mobile proxies (datacenter палится моментально)
    5. Tests: account.get("proxy") корректно используется в connect-flow
  • Зависимости: G2 (accounts.json) — done
  • Эффект: x3-5 к лайфтайму при наличии 2+ аккаунтов

A2. Дневной budget per account [M]

  • Файлы: новый daily_budget.py (или внутри account_state.py),
    bot.run_thread, avito_client.py, commercial_parser.py
  • Проблема: цикл while True. Аккаунт может за день сделать тысячи
    запросов "Показать телефон" — для реального пользователя это нонсенс.
  • Шаги:
    1. На базе E2 metrics (уже есть в БД) сделать guard'ы:
       • listings_parsed > 80/день  → стоп до завтра
       • messages_sent  > 30/день   → стоп messenger до завтра
       • get_phone      > 25/день   → стоп клики, остальные части OK
    2. API: account_state.check_daily_budget(name, action) -> bool /
       account_state.remaining_budget(name, action) -> int.
       Лимиты — глобальный default + per-account override в accounts.json.
    3. Хук в AvitoClient: перед find_and_view_..._listings проверяем
       listings budget; перед process_messages — messages budget.
    4. Логи: при упирании в budget — INFO в TG, чтобы было понятно почему
       бот замолчал.
    5. Tests: при listings_parsed=N, check_daily_budget("acc1", "listings")
       возвращает False если N >= limit.
  • Зависимости: E2 (metrics) — done
  • Эффект: x2-3 к лайфтайму — устраняет самый палевный признак "это бот"

A3. Лимит на "Показать телефон" [S]

  • Файл: commercial_parser.py:extract_listing_data (блок get_phone)
  • Проблема: клик на каждом листинге. У реального пользователя 1-2 раза
    в день максимум, у нас — на каждом из 5 листингов × N запусков в час.
    Это #1 сигнал анти-фрода.
  • Шаги:
    1. Дневной hard-limit (см. A2): >25/день → не кликаем.
    2. Сессионный soft-limit: в одной session не больше 30% листингов
       с кликом телефона. Реализация: random.random() < 0.3.
    3. Если в predыдущей session уже было >5 кликов — пропускаем телефон
       до конца текущей session.
    4. Cooldown после каждого клика 30-90 сек (а не 1-2 сек как сейчас).
    5. Hookup новой метрики E2: "phone_clicks" — для аналитики
  • Зависимости: A2 (или хотя бы простой счётчик в account_state)
  • Эффект: x2 к лайфтайму, минимум усилий

A4. Сессионные паузы между прогонами [S]

  • Файл: bot.run_thread (loop), tg_bot.py (для отображения "in cooldown")
  • Проблема: после messenger сразу новый search-цикл. Реальный
    пользователь делает паузы часами.
  • Шаги:
    1. После одного полного прохода (search + messenger) → пауза
       random.uniform(30, 90) минут.
    2. Длинная пауза должна реагировать на stop_event (использовать
       human_delay/wait с проверкой через каждые ~10 сек).
    3. Лог в TG: "[acc1] завершил цикл, следующий в ~HH:MM" —
       так оператор видит что бот не висит.
    4. Параметризация: session_pause_min / session_pause_max в
       accounts.json (per-account) с глобальным default'ом из config.json.
  • Зависимости: нет
  • Эффект: x1.5-2 к лайфтайму

БЛОК B — TIER 2: ДЛЯ ДОЛГОЖИТЕЛЕЙ (1-3 месяца лайфтайма)

B1. Warmup-режим для новых аккаунтов [M]

  • Файлы: новый flag-поле в БД (или в accounts.json), bot.run_thread,
    avito_client.py
  • Проблема: новый аккаунт без истории сразу начинает клацать "Показать
    телефон" → красный флаг для антифрода с первого дня.
  • Шаги:
    1. В accounts.json добавить поле "created_at": "2025-01-15" или
       флаг "warmup": true.
    2. В первые N дней (default 3) после created_at:
       • НЕ кликать "Показать телефон" (вообще)
       • НЕ отправлять сообщения через LLM (только просмотр)
       • Парсить меньше — 20 листингов/день максимум
       • Больше времени на каждый листинг (читаем описание дольше)
       • Скроллим случайные категории, заходим в "Профиль"/"Избранное"
    3. Через N дней автоматически переключение в normal режим
       (лог + опционально уведомление в TG).
    4. Команда /warmup acc1 5 в TG — продлить warmup ещё на 5 дней.
  • Зависимости: A2 (дневные лимиты), A3 (контроль get_phone)
  • Эффект: x3 к лайфтайму для новых аккаунтов

B2. Активное окно времени [S]

  • Файл: bot.run_thread, account_state.py
  • Проблема: бот парсит в 3 ночи так же, как в 14:00. Реальные
    пользователи спят.
  • Шаги:
    1. В config.json (+ per-account override): active_hours_start / end,
       например 9..23 по UTC+3 (default).
    2. В run_thread перед каждым циклом проверять — если сейчас вне окна,
       спать до начала окна (но реагировать на stop_event).
    3. Лог: "[acc1] вне активного окна — сплю до 09:00".
    4. Можно сделать "soft mode" в нерабочее время — только messenger
       (отвечать тем, кто пишет ночью), без парсинга.
  • Зависимости: нет
  • Эффект: x1.5 к лайфтайму, плюс снижение нагрузки на прокси/сервер

B3. Рандомизация старта потоков [S]

  • Файл: bot._launch_commercial_bot_threads
  • Проблема: time.sleep(5) между стартами потоков — слишком регулярный
    паттерн (все аккаунты заходят в "минута=5, минута=10" каждое утро).
  • Шаги:
    1. Заменить time.sleep(5) на human_delay(30, 180) — случайные 0.5-3 мин.
    2. Опционально: первый запуск дня каждый аккаунт стартует в случайные
       минуты часа (не все ровно в HH:00).
  • Зависимости: нет
  • Эффект: x1.2 (мелкое улучшение, но почти бесплатное)

B4. Long-cooldown при множественных капчах [S]

  • Файл: account_state.py
  • Проблема: после 3-5 капч за день аккаунт уже под подозрением.
    Текущий 30-мин cooldown не помогает — на следующий час мы получим
    ещё капчу, потом ещё, и так до бана.
  • Шаги:
    1. В account_state.mark_captcha считать N капч за последние 24h
       (или брать из E2 metrics — там есть captcha_hits).
    2. Если N >= 3 → cooldown 4-8 часов (не минут).
    3. Если N >= 5 → cooldown до следующего календарного дня + сильно
       снизить дневной budget (см. A2) на ~50%.
    4. Алерт в TG: "[acc1] 3 капчи за сутки — длинный cooldown".
  • Зависимости: A2 (для budget reduction), E2 — done
  • Эффект: спасает аккаунт после первой серьёзной "красной зоны"

БЛОК D — МАСШТАБИРОВАНИЕ (только при росте до 10+ аккаунтов)

D1. G3: SQLite task queue [L]

  • Файлы: database.py (новая таблица tasks), новый scheduler.py
  • Проблема: каждый поток крутит свой бесконечный цикл. При 10+
    аккаунтах сложно балансировать нагрузку, retry/backoff, priority.
  • Шаги:
    1. Таблица tasks(id, account_name, kind, payload, status,
       attempts, next_attempt_at, created_at).
    2. Worker-pool по аккаунтам читает свои задачи из таблицы.
    3. Retry с exponential backoff: 1m → 5m → 15m → 60m → dead.
    4. Priority queue: messenger > listings (новые сообщения важнее).
  • Зависимости: A2 (без budget'а нет смысла приоритезировать), C1
  • Эффект: операционно (видимость) — нужно ТОЛЬКО при росте.
    На 1-5 аккаунтах текущая архитектура справляется.
  • РЕКОМЕНДАЦИЯ: НЕ ДЕЛАТЬ сейчас. Зафиксировать как идею.
