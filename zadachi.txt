КОНТЕКСТ И ЦЕЛЬ
================

Текущее состояние: бот функционально готов. Sprints 1-5 + E2(metrics) + E2(search_filters) +
G1 + G2 + Tier1(A1-A4) + Tier2(B1-B4) + Tier3(C1-C3) done.
187 тестов проходят, ruff чистый. Селениум-флоу за единым фасадом
AvitoClient, метрики per account / per hour работают, accounts.json вынесен,
per-account search_filters реализованы, защита аккаунтов (proxy, budget,
phone-limit, session-паузы, warmup, active-hours, health-score, captcha-log)
полностью реализована и интегрирована.

ТЕКУЩАЯ ЦЕЛЬ: максимизировать лайфтайм аккаунтов (3-6+ мес/аккаунт).
Прямая дорога — БЛОК F (Tier 4: Behavioral Fingerprinting), 12 задач,
описанных в деталях для делегирования. См. секцию РЕКОМЕНДУЕМЫЙ ПОРЯДОК
снизу — задачи отсортированы по соотношению эффект/усилие.

Реальная проблема СЕЙЧАС — лайфтайм Avito-аккаунтов. Без описанных ниже
доработок реалистичная вилка:

  • proxies.txt пуст (общий IP)        → 3-7 дней до капчи, 1-2 нед до бана
  • резидентский proxy per account     → 2-3 недели до проблем, 1-2 мес до бана
  • + дневные лимиты + warmup          → 1-3 мес до проблем, 3-6+ мес до бана

ВЫВОД: главный приоритет сейчас — не новые фичи, а защита аккаунтов.

Что УЖЕ работает в защиту:
  • AdsPower (antidetect fingerprinting) — главное
  • B2 human-delays (normal distribution в ключевых точках)
  • B4 native session reuse (после первого логина не дёргаем /login)
  • A3 captcha-cooldown 30 мин (per-account override через G2)
  • B1 SMS/captcha обработка через TG-резюм
  • Random sample(5) + random city/category/deal_type на каждую итерацию
  • Yandex warmup (не заходим прямо на Avito)
  • E2 metrics — есть на чём строить дневные лимиты

Что ТЕПЕРЬ защищено (было проблемой, теперь реализовано):
  ✓ per-account proxy (A1) — каждый аккаунт через свой прокси
  ✓ Дневные лимиты listings/messages/phone (A2) — guard'ы в AvitoClient
  ✓ Клик "Показать телефон" ограничен (A3) — 30% soft-limit + дневной хард
  ✓ Паузы 30-90 мин между прогонами (A4) — session_pause_min/max
  ✓ Warmup для новых аккаунтов (B1) — /warmup команда в TG
  ✓ Активное окно времени (B2) — active_hours_start/end, per-account
  ✓ Рандомизация старта потоков (B3) — 30-180 сек между запусками
  ✓ Long-cooldown при множественных капчах (B4) — 4-8ч при ≥3, до следующего дня при ≥5
  ✓ Budget alerts 80%/100% в TG (C2) — de-dup per день
  ✓ Captcha log с traceability (C3) — /lastcaptcha команда в TG
  ✓ Per-account health score (C1) — /health команда, автоснижение бюджета

  • Файл: commercial_parser.py:extract_listing_data (блок get_phone)
  • Проблема: клик на каждом листинге. У реального пользователя 1-2 раза

    3. Если в predыдущей session уже было >5 кликов — пропускаем телефон
       до конца текущей session.
    4. Cooldown после каждого клика 30-90 сек (а не 1-2 сек как сейчас).
    5. Hookup новой метрики E2: "phone_clicks" — для аналитики

БЛОК F — TIER 4: BEHAVIORAL FINGERPRINTING (3-6+ мес. лайфтайма)

Эти задачи касаются паттернов поведения, которые антифрод видит даже
при идеальной защите аккаунтов (прокси + AdsPower + лимиты). Цель —
сделать действия бота статистически неотличимыми от живого пользователя.
Не критично при наличии Tier 1-3, но даёт +30-50% к лайфтайму при долгой
эксплуатации (3+ месяцев на аккаунт).

F2. Variable batch sizes (листинги и категории per cycle) [S]

  • Файлы:
      - bot.py:594-640 (browse_commercial_categories) — параметр
        num_categories=3, ads_per_category=3.
      - bot.py:642-735 (find_and_view_commercial_listings) — захардкожено
        `random.sample(links, min(5, len(links)))` (стр. 696).
  • Проблема: каждый цикл бот парсит:
      - В browse: РОВНО 3 категории × РОВНО 3 листинга (зашитые defaults)
      - В find_and_view: РОВНО 5 листингов
    Антифрод видит в логах паттерн: 11:00 — 5 listings, 13:00 — 5 listings,
    14:30 — 5 listings, ... Это очень регулярно. Реальный пользователь
    смотрит то 2 листинга и закрывает (надоело), то 8 (заинтересовался).

  • Шаги:
    1. В bot.py:642-735 заменить `random.sample(links, min(5, len(links)))`
       на использование функции с весовым распределением:
       ```python
       def _weighted_listing_count(max_n: int = 7) -> int:
           # Вес: чаще 2-4, реже 1, 5-7. Вес 0 = практически невозможно.
           weights = [0, 0.10, 0.25, 0.30, 0.20, 0.10, 0.04, 0.01]  # idx=0..7
           weights = weights[:max_n + 1]
           return random.choices(range(len(weights)), weights=weights)[0]
       ```
       Положить в bot.py рядом с view_listing (или в новый
